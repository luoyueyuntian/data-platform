# PRD V1.1 — 核心分析增强

## 1. 版本目标

在 V1.0 的基础上补齐漏斗、留存、分布三大核心分析模型，并支持分析结果保存、复用与导出，让平台具备真正可用的分析中台雏形。

## 2. 目标用户与价值

| 用户 | 版本价值 |
|------|----------|
| 产品经理 | 可识别关键转化链路的流失位置 |
| 增长分析师 | 可判断渠道/版本对留存和分布的影响 |
| 业务负责人 | 可把高频分析沉淀为书签和导出结果 |

## 3. 功能范围

| 模块 | 子能力 | 优先级 |
|------|--------|--------|
| 漏斗分析 | 多步骤漏斗、转化率、流失率、时间窗口、趋势 | P0 |
| 留存分析 | 首次行为、回访行为、自定义周期、留存表、留存曲线 | P0 |
| 分布分析 | 次数分布、金额分布、分桶、统计指标 | P0 |
| 分析保存 | 书签、复制分析、导出图片/CSV | P1 |

## 4. 明确不在本版本范围内

- 路径、会话、用户画像、用户分群。
- 标签、触达、实验。
- 统一指标平台与报表订阅。

## 5. 功能详细设计

### 5.1 漏斗分析

#### 5.1.1 场景定义

用于分析用户在一组有序行为步骤中的转化情况，例如：
- 浏览商品 -> 加入购物车 -> 提交订单 -> 支付成功。
- 访问注册页 -> 提交注册 -> 完成引导。

#### 5.1.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| FUN-001 | 漏斗配置 | 最少 2 步，最多 8 步 |
| FUN-002 | 步骤筛选 | 每一步支持单独配置事件属性筛选 |
| FUN-003 | 时间窗口 | 支持 1 小时、1 天、7 天、30 天、自定义 |
| FUN-004 | 转化模式 | 首次转化、任意转化 |
| FUN-005 | 趋势分析 | 支持观察总体转化率随时间变化 |
| FUN-006 | 分组对比 | 支持按渠道、端、地区等属性分组 |

#### 5.1.3 核心规则

- 同一用户只有按时间顺序完成步骤，才视为转化。
- 超过漏斗窗口的行为不计入转化链路。
- 首次转化模式下，同一用户只计算第一条完成链路。
- 任意转化模式下，同一用户可重复进入漏斗。

#### 5.1.4 核心接口

`POST /api/v1/analysis/funnels`

```json
{
  "project_id": "proj_001",
  "name": "下单转化漏斗",
  "steps": [
    { "event_name": "product_view" },
    { "event_name": "add_to_cart" },
    { "event_name": "order_created" },
    { "event_name": "order_paid" }
  ],
  "conversion_mode": "first",
  "window": { "value": 7, "unit": "day" },
  "group_by": ["channel"],
  "date_range": { "start": "2025-04-01", "end": "2025-04-30" }
}
```

### 5.2 留存分析

#### 5.2.1 场景定义

用于分析某批用户在首个行为发生后，后续是否持续回访或重复完成某类行为。

#### 5.2.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| RET-001 | 首次行为定义 | 支持注册、登录、下单等事件作为起始行为 |
| RET-002 | 回访行为定义 | 支持“任意活跃”或指定事件回访 |
| RET-003 | 留存周期 | 按日、周、月计算 |
| RET-004 | 观察窗口 | 默认 7、14、30、60、90 天 |
| RET-005 | 留存表 | 展示分 cohort 的留存矩阵 |
| RET-006 | 留存曲线 | 展示总体或分组留存趋势 |

#### 5.2.3 核心规则

- 留存用户必须先进入起始 cohort。
- Day N 留存的判断依据为第 N 个时间周期内是否完成回访条件。
- 同一 cohort 的基数在整个留存表中保持不变。

#### 5.2.4 核心接口

`POST /api/v1/analysis/retention`

```json
{
  "project_id": "proj_001",
  "first_event": "sign_up",
  "return_event": "app_open",
  "period": "day",
  "window": 30,
  "group_by": ["channel"],
  "date_range": { "start": "2025-04-01", "end": "2025-04-30" }
}
```

### 5.3 分布分析

#### 5.3.1 场景定义

用于观察某个数值型或次数型指标在用户集合中的分布情况，例如：
- 每个用户近 30 天购买次数分布。
- 每个订单的支付金额分布。

#### 5.3.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| DIS-001 | 次数分布 | 按用户统计事件次数分布 |
| DIS-002 | 属性分布 | 按数值属性统计分布 |
| DIS-003 | 分桶方式 | 自动分桶、自定义区间分桶 |
| DIS-004 | 统计指标 | 均值、中位数、P90、P99 |
| DIS-005 | 分组 | 支持按用户属性进行分组对比 |

#### 5.3.3 核心规则

- 分布分析默认作用于“用户集合”或“事件集合”，查询前必须明确对象。
- 数值分布只接受数值型属性。
- 区间必须完整覆盖所有值，超出部分进入“其他”桶。

### 5.4 分析保存与导出

#### 5.4.1 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| SAVE-001 | 保存书签 | 保存完整分析配置 |
| SAVE-002 | 复制分析 | 基于既有分析复制出新版本 |
| SAVE-003 | 导出图片 | 导出当前图表快照 |
| SAVE-004 | 导出数据 | 导出当前明细表为 CSV |

#### 5.4.2 规则

- 书签保存的是配置，不是静态结果。
- 当依赖事件或属性被停用时，书签需给出兼容提示。

## 6. 跨模块关系

- 漏斗、留存、分布都可保存为书签。
- 书签可在后续版本被看板、报表或资产包复用。

## 7. 验收标准

### 7.1 功能验收

- 漏斗支持 2 至 8 步步骤配置。
- 漏斗支持首次转化与任意转化两种模式。
- 留存支持起始行为、回访行为和按日/周/月周期分析。
- 分布支持次数分布和数值属性分布。
- 分析结果可保存为书签并再次打开。
- 图表和明细支持导出。

### 7.2 数据正确性验收

- 漏斗步骤转化率、总体转化率和流失率计算正确。
- 留存 cohort 基数固定，不得随观察天数变化。
- 中位数、P90、P99 等统计指标与标准计算口径一致。

### 7.3 体验验收

- 漏斗和留存页面在复杂配置下仍能清楚展示关键口径说明。
- 导出文件字段顺序和页面明细字段保持一致。
