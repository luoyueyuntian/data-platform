# PRD V2.2 — 平台开放与 AI

## 1. 版本目标

让平台从“内部使用的分析系统”升级为“可被复用、可被集成、可被自然语言驱动的平台”。

本版本的重点是四件事：
- 统一指标口径。
- 建立正式报表层。
- 开放标准化 API 能力。
- 提供可控的 AI 辅助入口。

## 2. 目标用户与价值

| 用户 | 版本价值 |
|------|----------|
| 业务负责人 | 可通过统一指标和报表获取稳定经营视图 |
| 分析师 | 可做更灵活的 SQL 和指标建模 |
| 外部系统 | 可通过 OpenAPI 和 Webhook 接入分析结果 |
| 运营/产品团队 | 可通过 AI 更快完成圈人、问答和文案初稿 |

## 3. 功能范围

| 模块 | 子能力 | 优先级 |
|------|--------|--------|
| 指标平台 | 指标定义、分类、口径说明、维度关系、血缘 | P0 |
| 可视化报表 | 报表、组件联动、看板模式、订阅推送 | P0 |
| 自定义 SQL | SQL 编辑器、保存查询、结果可视化、权限控制 | P1 |
| OpenAPI | API Key、分析查询、受众判定、数据写入、Webhook | P0 |
| AI 助手 | AI 圈人、AI 分析问答、AI 文案、AI 诊断 | P1 |
| 国际化 | 多语言、项目时区、跨时区报表 | P1 |

## 4. 明确不在本版本范围内

- 无人值守的全自动运营策略执行。
- 面向开发者的插件市场。
- 复杂机器学习模型托管。

## 5. 功能详细设计

### 5.1 指标平台

#### 5.1.1 场景定义

指标平台用于解决“同一个业务指标在不同看板、报表、分析页面口径不一致”的问题。

#### 5.1.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| MET-001 | 指标定义 | 定义名称、描述、单位、负责人 |
| MET-002 | 计算规则 | 定义来源事件、过滤条件、聚合方式、公式 |
| MET-003 | 维度关系 | 指定指标可按哪些维度切分 |
| MET-004 | 分类管理 | 按业务域组织指标 |
| MET-005 | 指标血缘 | 展示指标依赖关系 |
| MET-006 | 指标分析 | 直接基于指标做趋势与对比 |

#### 5.1.3 指标定义示例

```json
{
  "name": "paying_users",
  "display_name": "支付用户数",
  "category": "transaction",
  "description": "统计周期内完成支付的去重用户数",
  "unit": "人",
  "calculation": {
    "source": "event",
    "event_name": "order_paid",
    "metric": "unique_users"
  },
  "dimensions": ["channel", "device_type", "region"]
}
```

### 5.2 可视化报表

#### 5.2.1 场景定义

报表层面向稳定经营和周期复盘，强调组件编排、统一筛选和订阅分发。

#### 5.2.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| REP-001 | 报表创建 | 空白报表或模板报表 |
| REP-002 | 数据源 | 支持分析模型结果和指标平台指标 |
| REP-003 | 组件编排 | 折线、柱状、表格、漏斗、指标卡等 |
| REP-004 | 联动过滤 | 点击组件维度值影响其他组件 |
| REP-005 | 订阅推送 | 邮件、企业微信、Webhook |
| REP-006 | 看板模式 | 全屏轮播和大屏展示 |

#### 5.2.3 核心规则

- 报表级筛选条件必须一处配置、全局生效。
- 组件引用的指标版本需可追踪。
- 订阅失败要有记录和补发机制。

### 5.3 自定义 SQL

#### 5.3.1 场景定义

用于满足高级分析师的探索性查询需求，但必须确保权限、资源和安全边界可控。

#### 5.3.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| SQL-001 | 编辑器 | 语法高亮、格式化、自动补全 |
| SQL-002 | 查询执行 | 异步执行，查看进度与结果 |
| SQL-003 | 保存查询 | 复用 SQL 模板和查询历史 |
| SQL-004 | 结果可视化 | 一键生成表格或基础图表 |
| SQL-005 | 安全控制 | 只读、超时、限流、结果行数限制 |

#### 5.3.3 核心规则

- 仅支持只读查询。
- 超时和结果上限必须明确展示。
- 查询权限需受项目和角色控制。

### 5.4 OpenAPI 与 Webhook

#### 5.4.1 场景定义

对外提供标准化数据能力，让外部系统可以安全读取分析结果或写入指定数据。

#### 5.4.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| OPEN-001 | API Key 管理 | 创建、禁用、过期控制、权限范围 |
| OPEN-002 | 分析查询 API | 查询事件分析、漏斗、留存、指标结果 |
| OPEN-003 | 受众判定 API | 判断用户是否属于某个受众或分群 |
| OPEN-004 | 数据写入 API | 写入事件数据或用户属性 |
| OPEN-005 | Webhook | 报表订阅、预警通知、任务回调 |
| OPEN-006 | API 文档 | 在线文档、示例代码、错误码说明 |

#### 5.4.3 核心规则

- API Key 必须绑定项目和权限范围。
- 分析查询类接口默认限频。
- 高风险接口必须记录调用审计日志。

### 5.5 AI 助手

#### 5.5.1 场景定义

AI 助手不是替代规则系统，而是降低规则配置和分析理解门槛。

#### 5.5.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| AI-001 | AI 圈人 | 将自然语言转为分群规则草稿 |
| AI-002 | AI 问答 | 基于已有分析能力回答趋势和对比问题 |
| AI-003 | AI 文案 | 为触达计划生成文案初稿 |
| AI-004 | AI 诊断 | 对异常指标给出可能原因和建议动作 |

#### 5.5.3 核心规则

- AI 输出默认是“建议”，不是直接生效配置。
- 涉及创建分群、发送触达、修改指标等动作时必须二次确认。
- AI 回答应引用其使用的数据范围和时间范围。

### 5.6 国际化与时区

#### 5.6.1 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| I18N-001 | 多语言界面 | 中英文切换 |
| I18N-002 | 项目时区 | 各项目独立配置时区 |
| I18N-003 | 报表时区 | 报表和订阅按项目时区计算 |
| I18N-004 | 时间展示 | 页面统一展示时间来源和时区 |

## 6. 跨模块关系

- 指标平台为报表、预警、OpenAPI 和 AI 问答提供统一口径。
- 自定义 SQL 可将查询保存为书签或报表组件数据源。
- OpenAPI 复用平台已有查询能力，不单独创造第二套口径。
- AI 圈人复用分群中心，AI 问答复用分析和指标平台。

## 7. 验收标准

### 7.1 功能验收

- 指标支持定义、分类、搜索、复用。
- 报表支持组件编排、联动过滤和订阅推送。
- 自定义 SQL 支持执行、保存、结果可视化。
- OpenAPI 支持 API Key 管理、查询、受众判定和 Webhook。
- AI 助手可生成分群规则草稿、分析回答和文案草稿。
- 多语言和多时区配置可正常生效。

### 7.2 数据正确性验收

- 同一指标在分析页、报表页、订阅结果中的数值一致。
- 受众判定 API 与平台内分群结果一致。
- AI 问答使用的时间范围和指标口径可追溯。

### 7.3 平台化验收

- API Key 能按范围授权并正确失效。
- 订阅和 Webhook 调用结果可追踪。
- SQL 权限边界清晰，越权查询可被阻止并提示原因。
