# PRD 总览：优化版用户行为分析平台

## 1. 产品愿景

构建一款面向企业的用户行为分析与智能运营平台，覆盖从数据接入、行为分析、用户洞察到运营闭环、数据治理、开放集成和 AI 辅助的完整链路。

产品目标不是机械复刻单一竞品，而是沉淀一套更易落地、更易演进的能力版图：
- 先完成核心分析闭环。
- 再建立用户洞察与运营闭环。
- 最后完成治理、平台化和智能化能力。

## 2. 目标用户

| 用户角色 | 核心问题 | 主要使用能力 |
|----------|----------|--------------|
| 产品经理 | 功能是否被使用、转化卡在哪里 | 事件分析、漏斗、留存、路径 |
| 数据分析师 | 如何快速完成行为分析和专项诊断 | 事件分析、分布、会话、归因、SQL |
| 运营人员 | 如何找到人、触达人、评估效果 | 分群、标签、生命周期、触达、A/B |
| 增长团队 | 渠道和实验的真实效果是什么 | 归因、实验、预警、旅程 |
| 管理员 | 如何统一数据规则并控制权限风险 | 数据治理、维度表、虚拟事件、RBAC |
| 外部系统集成方 | 如何安全获取分析结果或写入数据 | OpenAPI、API Key、Webhook |

## 3. 产品设计原则

### 3.1 先统一数据语义，再扩张分析能力

如果事件、用户、属性、指标口径不统一，后续任何分析模型都会失真。

### 3.2 先做高频能力，再做重型能力

优先完成最常用的事件分析、漏斗、留存、分群、看板，再进入指标平台、AI、开放生态等重型模块。

### 3.3 同一能力只定义一个权威入口

例如：
- 用户分群由统一分群中心定义。
- 标签由统一标签中心管理。
- 指标由统一指标平台定义。
- OpenAPI 权限由统一 API Key 机制管理。

### 3.4 产品规则要稳定、可复用、可解释

每个分析模型都需要明确：
- 输入对象是什么。
- 计算口径是什么。
- 结果是否实时。
- 是否能保存、复用、下发到其他模块。

## 5. 版本规划

### V1.0 — 数据接入与基础分析

**版本目标**：完成数据采集、元数据管理、事件分析、看板和项目基础能力，建立最小可用闭环。

**不含内容**：漏斗、留存、路径、分群、运营触达。

### V1.1 — 核心分析增强

**版本目标**：补齐分析型产品最常用的三大模型：漏斗、留存、分布，并提供分析保存与导出能力。

**不含内容**：用户分群、标签、触达、实验。

### V1.2 — 用户洞察与用户分群

**版本目标**：把分析对象从“事件结果”扩展为“用户行为过程与用户集合”。

**不含内容**：营销触达、实验、治理、开放平台。

### V2.0 — 智能运营闭环

**版本目标**：完成从标签、受众、生命周期到触达与实验的运营闭环。

**不含内容**：多项目治理、指标平台、OpenAPI、AI 助手。

### V2.1 — 数据治理与企业能力

**版本目标**：为复杂企业场景提供数据治理、企业权限与资产复用能力，并补齐归因与预警。

**不含内容**：统一指标平台、开放查询平台、AI 助手。

### V2.2 — 平台开放与 AI

**版本目标**：让平台具备统一指标、可视化报表、开放能力和自然语言辅助能力。

## 6. 通用业务对象定义

### 6.1 事件

事件是用户在某一时间点发生的一次行为记录。

核心字段：
- `event_name`：事件名。
- `time`：事件发生时间。
- `distinct_id`：匿名或设备标识。
- `user_id`：登录用户标识。
- `properties`：事件属性。
- `project_id`：所属项目。

### 6.2 用户

用户是平台进行分析、分群和运营的核心对象。

平台必须同时支持：
- 匿名用户。
- 登录用户。
- 匿名与登录身份合并。

### 6.3 属性

属性分为三类：
- 事件属性：描述一次行为。
- 用户属性：描述一个用户。
- 系统预置属性：平台自动采集或自动补充。

### 6.4 分群

分群是满足一组规则条件的用户集合，可被其他模块复用。

### 6.5 标签

标签是写回到用户维度的特征值，可作为分析维度或运营条件。

### 6.6 指标

指标是平台内被统一定义和复用的业务口径，例如 DAU、支付人数、支付金额、转化率。

## 7. 通用约定

### 7.1 用户标识规则

- 登录用户以 `user_id` 为主标识。
- 匿名用户以 `distinct_id` 为主标识。
- 平台需要支持身份合并，将匿名行为归属到登录用户。
- 所有分析都必须清楚说明是按 `用户` 还是按 `设备/匿名标识` 统计。

### 7.2 时间规则

- 所有分析都基于项目时区计算。
- 默认支持按小时、天、周、月聚合。
- 涉及窗口计算的模型必须明确窗口口径，例如 30 分钟会话、7 天归因、30 天活跃等。

### 7.3 权限规则

基础角色：
- `owner`：项目所有者。
- `admin`：项目管理员。
- `analyst`：分析师。
- `operator`：运营人员。
- `viewer`：查看者。

### 7.4 API 响应格式

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "req_xxx"
}
```

### 7.5 页面交互约定

所有分析型页面建议统一为三段式：
- 条件配置区。
- 图表结果区。
- 明细/导出/保存区。

## 8. 通用非功能性要求

| 指标 | 目标 |
|------|------|
| 数据接入成功率 | >= 99.9% |
| 核心分析查询成功率 | >= 99.5% |
| 事件分析查询响应 | P95 < 3 秒 |
| 漏斗/留存/路径等重型分析 | P95 < 8 秒 |
| 预警触发延迟 | 规则满足后 5 分钟内 |
| 报表订阅准点率 | >= 99% |

## 9. 跨版本复用关系

| 上游能力 | 被下游模块复用方式 |
|----------|--------------------|
| 元事件与属性定义 | 被所有分析和分群模块引用 |
| 事件分析查询能力 | 被看板、报表、预警、OpenAPI 复用 |
| 用户分群 | 被受众、触达、实验、AI 圈人复用 |
| 标签体系 | 被画像、受众、触达、报表、AI 复用 |
| 指标平台 | 被报表、预警、OpenAPI、AI 问答复用 |
