# PRD V1.2 — 用户洞察与用户分群

## 1. 版本目标

把平台从“看事件结果”升级为“看用户行为过程与用户集合”。

本版本需要回答四类问题：
- 用户通常怎么走到关键行为。
- 一次会话里发生了什么。
- 某个用户当前是什么状态。
- 哪些用户满足某一类运营或分析条件。

## 2. 目标用户与价值

| 用户 | 版本价值 |
|------|----------|
| 产品经理 | 识别关键路径、跳出点和行为串联问题 |
| 运营人员 | 找到目标用户并形成可复用用户群 |
| 分析师 | 从“统计结果”深入到“用户层级洞察” |

## 3. 功能范围

| 模块 | 子能力 | 优先级 |
|------|--------|--------|
| 路径分析 | 起点/终点限制、步数限制、路径明细、分组 | P0 |
| 会话分析 | 会话定义、时长、深度、频次、会话明细 | P0 |
| 用户画像 | 用户列表、详情、属性卡片、事件时间线 | P0 |
| 用户分群 | 属性规则、行为规则、AND/OR 条件组、预估人数、静态/动态分群 | P0 |
| 分群复用 | 分群导出、应用到分析模块 | P1 |

## 4. 明确不在本版本范围内

- 标签写回、触达计划、A/B 测试。
- 虚拟事件、维度表、指标平台、AI 圈人。

## 5. 功能详细设计

### 5.1 路径分析

#### 5.1.1 场景定义

路径分析用于查看用户在同一会话或同一分析窗口内的行为流转路径，帮助发现：
- 最常见的成功路径。
- 异常回路和流失路径。
- 起始页或关键动作后的分流去向。

#### 5.1.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| PATH-001 | 分析模式 | 从某事件开始、到某事件结束、全路径 |
| PATH-002 | 步数限制 | 默认 5 步，最大 8 步 |
| PATH-003 | 事件排除 | 可排除噪音事件，如心跳、离开页 |
| PATH-004 | 分组对比 | 按端、渠道、地区等维度对比路径 |
| PATH-005 | 路径明细 | 展示 Top 路径、人数、占比、平均耗时 |

#### 5.1.3 核心规则

- 路径按时间顺序生成。
- 默认以会话为边界，跨会话行为不拼接到同一路径。
- 同一路径的统计口径为去重用户数或去重会话数，需在页面明确展示。

#### 5.1.4 核心接口

`POST /api/v1/analysis/path`

```json
{
  "project_id": "proj_001",
  "analysis_type": "start_from",
  "start_event": "$pageview",
  "max_steps": 5,
  "session_timeout_minutes": 30,
  "exclude_events": ["$pageleave", "$heartbeat"],
  "group_by": ["channel"],
  "date_range": { "start": "2025-04-01", "end": "2025-04-07" }
}
```

### 5.2 会话分析

#### 5.2.1 场景定义

会话分析用于衡量用户一次访问或一次使用过程的质量，包括访问时长、行为深度和回访频次。

#### 5.2.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| SES-001 | 会话定义 | 默认 30 分钟无操作视为新会话，支持自定义 |
| SES-002 | 会话指标 | 会话数、平均时长、平均深度、跳出率 |
| SES-003 | 分布分析 | 会话时长分布、深度分布 |
| SES-004 | 会话明细 | 查看单次会话内事件序列 |
| SES-005 | 分组对比 | 支持按渠道、端、用户属性分组 |

#### 5.2.3 核心规则

- 跳出会话定义为仅包含 1 个关键行为的会话。
- 会话深度指单次会话中的事件数。
- 会话指标必须说明统计对象是“会话”而非“用户”。

### 5.3 用户画像

#### 5.3.1 场景定义

用户画像模块用于完成用户级排查和运营前洞察，是“从图表下钻到单个用户”的入口。

#### 5.3.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| PROF-001 | 用户列表 | 支持按属性、分群、最近活跃时间筛选 |
| PROF-002 | 用户详情 | 展示用户基础信息、最新属性、最近行为 |
| PROF-003 | 事件时间线 | 按时间倒序展示事件序列 |
| PROF-004 | 行为摘要 | 展示近 7 天、30 天关键行为统计 |
| PROF-005 | 快速下钻 | 从分析结果点击用户群体进入用户列表 |

#### 5.3.3 页面规则

用户详情页建议分为三栏：
- 基础信息与属性卡片。
- 行为摘要和关键指标。
- 事件时间线。

### 5.4 用户分群

#### 5.4.1 场景定义

用户分群是后续所有运营动作的基础能力，同时也是分析过滤的统一入口。

#### 5.4.2 核心能力

| 需求 ID | 功能 | 描述 |
|---------|------|------|
| SEG-001 | 属性规则 | 支持基于用户属性做条件筛选 |
| SEG-002 | 行为规则 | 支持按事件次数、最近发生时间、是否完成某行为筛选 |
| SEG-003 | 条件组 | 支持多组 AND/OR 组合 |
| SEG-004 | 预估人数 | 保存前先估算人群规模 |
| SEG-005 | 分群类型 | 静态分群、动态分群 |
| SEG-006 | 分群详情 | 查看分群人数趋势和样本用户 |
| SEG-007 | 分群复用 | 分群可作为其他分析模块筛选条件 |

#### 5.4.3 核心规则

- 涉及行为条件的动态分群可以采用准实时或定时更新，但页面必须明确刷新频率。
- 保存分群时必须展示预估人数和最近更新时间。
- 分群规则修改后，需保留历史版本说明，避免误覆盖正在使用的人群。

#### 5.4.4 分群表达示例

```json
{
  "name": "近30天高价值用户",
  "type": "dynamic",
  "logic": "AND",
  "conditions": [
    {
      "kind": "user_property",
      "property": "vip_level",
      "operator": ">=",
      "value": 3
    },
    {
      "kind": "event_aggregate",
      "event_name": "order_paid",
      "metric": "sum",
      "property": "amount",
      "operator": ">",
      "value": 1000,
      "window": { "value": 30, "unit": "day" }
    }
  ]
}
```

#### 5.4.5 核心接口

- `POST /api/v1/segments`
- `POST /api/v1/segments/preview`
- `GET /api/v1/segments/:id`
- `GET /api/v1/segments/:id/users`

## 6. 跨模块关系

- 路径和会话分析结果可下钻到用户画像。
- 用户分群可作为事件、漏斗、留存、路径、会话分析的筛选条件。
- 用户画像页可查看用户是否属于某个分群。

## 7. 验收标准

### 7.1 功能验收

- 路径分析支持起始/终止事件限制和步数配置。
- 会话分析支持自定义超时时间并输出核心会话指标。
- 用户列表支持按属性、活跃时间和分群筛选。
- 用户详情可展示属性卡片和事件时间线。
- 用户分群支持属性规则、行为规则和条件组组合。
- 分群可预估人数、保存并复用到其他分析模块。

### 7.2 数据正确性验收

- 同一用户跨会话行为不应被错误拼接为单一路径。
- 跳出率、平均会话深度等指标口径清晰且计算一致。
- 动态分群更新后，符合条件的新用户会进入分群，不再符合条件的用户会移出分群。

### 7.3 体验验收

- 从图表结果下钻到用户列表路径清晰。
- 分群配置页在复杂条件下仍可读、可编辑、可复核。
