# 神策数据平台功能概述

> 参考来源：神策数据官网 ([sensorsdata.cn](https://www.sensorsdata.cn)) 及帮助中心文档 ([manual.sensorsdata.cn](https://manual.sensorsdata.cn))

---

## 一、平台定位与核心理念

神策数据（Sensors Data）是国内专业的大数据分析与营销科技服务商，核心价值主张为 **"数字化客户经营"**。平台基于 **SDAF 运营框架** 构建产品矩阵：

```
Sense（感知）→ Decision（决策）→ Action（行动）→ Feedback（反馈）
```

即：采集感知用户行为 → 通过分析辅助决策 → 采取运营行动 → 回收效果数据形成闭环。

## 二、产品矩阵

神策数据的产品体系分为三大板块：

```
┌─────────────────────────────────────────────────────────┐
│                      神策数据平台                         │
├──────────────┬──────────────────┬───────────────────────┤
│   数界平台    │     分析云        │       营销云           │
│  (数据根基)   │   (分析决策)      │     (行动触达)         │
├──────────────┼──────────────────┼───────────────────────┤
│ 数据采集 SDK │ 用户行为分析      │ SCMS 活动营销          │
│ 数据治理 SDG │ 用户画像 CDP     │ SCRM 社交营销          │
│ 数据仓库     │ 指标平台         │ SLink 超级链接         │
│ 智能引擎     │ 经营分析         │ 智能推荐              │
│ 私有云平台   │ 广告分析         │ 用户旅程编排           │
│              │ A/B 测试         │ 消息推送 & 弹窗        │
│              │ 可视化看板       │                       │
└──────────────┴──────────────────┴───────────────────────┘
```

### 2.1 数界平台（数据根基）

提供从数据采集到存储、治理的全链路基础能力。

| 模块 | 功能说明 |
|------|---------|
| **数据采集 SDK** | 50+ 开源 SDK，覆盖 Web、iOS、Android、小程序、Flutter、React Native、Node.js、鸿蒙 OS 等全端 |
| **数据治理 SDG** | 提供"螺旋式"数据治理：关注源头 → 关注波动 → 全面掌控；包含 ID-Mapping 打通多端用户标识 |
| **数据仓库** | 基于 Event-User-Item 模型的扩展多实体模型，融合行为数据与业务经营数据 |
| **数据智能引擎** | 预测、预警、LookAlike、LTV 预测等算法能力 |
| **私有云平台** | 支持私有化部署 + SaaS，具备 1000+ 集群运维能力 |

### 2.2 分析云

核心分析决策层，提供 14 个分析模型及多维度分析能力。

| 模块 | 功能说明 |
|------|---------|
| **用户行为分析** | 14 个分析模型（事件分析、漏斗分析、留存分析、分布分析、归因分析、路径分析等） |
| **用户画像 CDP** | 完整的标签中台：标签导入、生产、管理、导出、应用；RFM 分层；用户群画像分析 |
| **指标平台** | 统一管理指标口径、维度定义、关联关系建模；支持四则运算自定义指标 |
| **经营分析** | 打通用户行为与经营数据，支持多实体关联（用户-商品、门店-商品等），多维下钻 |
| **广告分析** | 全渠道广告效果归因，对接 Google Ads、Meta Ads、Appsflyer，前后链路数据打通 |
| **A/B 测试** | 可视化试验编辑器，支持多人群试验、时间片轮转试验 |
| **可视化看板** | BI 报表类功能，支持门户级、企业级看板 |

### 2.3 营销云

行动触达层，实现多渠道、自动化的营销运营。

| 模块 | 功能说明 |
|------|---------|
| **SCMS 活动营销** | 活动创建、执行、管理、反馈、迭代自动化的一站式营销平台 |
| **SCRM 社交营销** | 社交营销助手，打通全域数据，支持微信生态（公众号、企业微信、小程序） |
| **SLink 超级链接** | 支持拉新促活场景的超级链接工具 |
| **智能推荐** | 基于深度学习的个性化推荐系统，实现"千人千面" |
| **用户旅程编排** | 流程画布：圈选用户 → 设定触发逻辑和通道 → 统计触发结果 |
| **消息推送 & 弹窗** | APP 在线弹窗、消息推送，多通道触达用户 |
| **神策客景** | To B 客户全生命周期分析和运营系统，客户健康度评估、流失预警 |

## 三、核心数据模型：Event-User-Item

神策的底层数据模型经历了从 **Event-User** 到 **Event-User-Item** 的演进。

### 3.1 Event（事件实体）

描述 _"某用户 在某个时间点 某个地方 以某种方式 完成了某个具体事情"_。

| 维度 | 字段 | 说明 |
|------|------|------|
| Who | `distinct_id` | 用户标识（匿名：设备 ID；登录：用户 ID） |
| When | `time` | 事件发生时间，精确到毫秒 |
| Where | `$ip` / `$city` / `$province` | 地理位置，自动解析或手动设置 |
| How | `$os`、`$model`、`$app_version` 等 | 设备信息、操作系统、应用版本等预置属性 |
| What | `event` + 自定义属性 | 事件名称 + 业务自定义字段 |

示例数据格式：

```json
{
  "distinct_id": "2b0a6f51a3cd6775",
  "time": 1434556935000,
  "type": "track",
  "event": "PageView",
  "properties": {
    "$ip": "180.79.35.65",
    "page_name": "网站首页",
    "url": "www.demo.com"
  }
}
```

### 3.2 User（用户实体）

记录用户的长期属性（Profile），通过以下接口进行管理：

- `profile_set`：设置用户属性（覆盖写入）
- `profile_set_once`：设置一次性属性（仅首次生效）
- `profile_increment`：数值属性累加
- `profile_append`：列表属性追加

### 3.3 Item（物品实体）

为解决 Event 属性不可变导致的历史回溯问题（如商品重新分类后历史事件的分类属性仍是旧的），引入 Item 实体。

使用方式：

| 方式 | 说明 | 特点 |
|------|------|------|
| 关联导入 | 埋点时通过 Item ID 将 Item 属性作为 Event 一部分导入 | 简化埋点，查询性能无损 |
| 实时 Join | 查询时动态关联 Item 表 | 灵活性高，查询性能略降 |

## 四、数据采集与 SDK

### 4.1 采集方式

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **代码埋点** | 手动插入采集代码，最灵活、控制力最强 | 核心业务逻辑和关键转化路径 |
| **可视化全埋点** | 图形界面圈选页面元素配置事件，无需修改业务代码 | 运营人员快速配置 |
| **全埋点（默认采集）** | SDK 自动采集基础行为（页面浏览、点击、启动等） | 基础数据采集，开箱即用 |

### 4.2 预置属性

SDK 自动采集的设备与环境属性：

```
$ip、$time、$app_version、$city、$manufacturer、
$model、$os、$os_version、$screen_width、$screen_height、
$browser、$browser_version、$network_type 等
```

### 4.3 数据接入流程

```
理解基本概念 → 获取事件设计方案 → 配置数据接入地址
→ 区分测试/正式环境 → 实施接入（埋点/导入历史数据）
→ 数据测试与验证 → 上线正式环境
```

## 五、核心分析模型

神策分析提供 14 个高度抽象的分析模型，以下为核心模型详解。

### 5.1 事件分析

对用户行为事件进行指标统计、属性分组和条件筛选。

- **指标**：总次数、用户数、人均次数、总和、均值、最大值、最小值、去重数
- **功能**：多维分组、自定义指标四则运算、时间对比（环比/同比）
- **图表**：线图、柱状图、饼图、累积图、表格

### 5.2 漏斗分析

分析多步骤过程中的转化与流失。

- **核心概念**：步骤（事件+筛选条件）、窗口期（完成时间限制）、属性关联
- **输出**：各步骤用户数、转化率、流失用户列表、转化时间中位数

### 5.3 留存分析

分析初始行为用户后续的持续参与情况。

- **功能**：初始行为与后续行为可相同或不同事件，支持 LTV 计算
- **应用**：衡量产品粘性、评估迭代效果、识别高忠诚度用户

### 5.4 分布分析

展示用户在特定指标下的结构化分段分布。

- **维度**：按次数统计、按时间段统计、按事件属性统计
- **优势**：优于单纯看 DAU，揭示单个用户对产品依赖程度的变化

### 5.5 归因分析

量化各营销触点对目标转化事件的贡献。

| 归因模型 | 分配方式 | 适用场景 |
|---------|---------|---------|
| 首次触点 | 第一个待归因事件功劳 100% | 重视新用户线索 |
| 末次触点 | 最后一个待归因事件功劳 100% | 电商站内归因 |
| 线性归因 | 平均分配功劳 | 坑位效果平均 |
| 位置归因 | 首尾各 40%，其余 20% | 兼顾线索与决策 |
| 时间衰减 | 越靠近转化贡献越大 | 重视近期触点 |

### 5.6 其他分析模型

- **路径分析**：用户行为路径轨迹
- **点击分析**：页面元素点击热度
- **间隔分析**：事件之间的时间间隔
- **属性分析**：用户属性分布统计
- **SQL 查询**：自定义 SQL 取数分析

## 六、技术架构

### 6.1 系统架构

```
客户端 SDK ─→ Nginx (HTTP API) ─→ Extractor ─→ Kafka
                                                    │
                                              Data Loader (ETL)
                                                    │
                                          ┌─────────┴──────────┐
                                          │    Kudu (WOS)      │
                                          │  实时写入层         │
                                          └─────────┬──────────┘
                                                    │ 批量合并 (Merger)
                                          ┌─────────┴──────────┐
                                          │  Parquet / HDFS    │
                                          │  (ROS 批量查询层)   │
                                          └─────────┬──────────┘
                                                    │
                                          ┌─────────┴──────────┐
                                          │  Impala (MPP 查询)  │
                                          └─────────┬──────────┘
                                                    │
                                              Web Server (API)
```

### 6.2 技术选型

| 组件 | 技术 |
|------|------|
| 数据接入 | Nginx + HTTP API |
| 消息队列 | Kafka |
| ETL | Data Loader |
| 批量计算 | Merger / Segmenter on YARN |
| 实时存储 | Kudu |
| 批量存储 | Parquet / HDFS |
| 查询引擎 | Impala (MPP) |
| 部署 | Kubernetes / K3s |

### 6.3 核心设计理念

- **ROLAP 架构**：非 MOLAP，存储最细粒度明细数据，支持任意维度下钻
- **不预定义指标**：查询时即时计算，灵活性最大化
- **追加为主**：事件数据追加写入，提供有限删除能力

## 七、部署方式

| 部署模式 | 说明 |
|---------|------|
| **私有化部署** | 数据部署在企业自己的服务器，支持 K8s/K3s 容器化 |
| **SaaS 部署** | 使用神策提供的云服务 |
| **混合部署** | 数据本地私有化 + 在线服务 SaaS 化 |

私有化经历了三个演进阶段：单产品线（2015-2018）→ 产品矩阵（2018-2019）→ 云操作系统/K8s（2020 至今）

## 八、权限管理

### 8.1 角色类型

| 类型 | 说明 |
|------|------|
| **系统角色** | 预置角色（管理员、分析师、开发者、普通用户），不可修改 |
| **自定义角色** | 用户创建，灵活配置权限点 |

### 8.2 权限维度

| 维度 | 说明 |
|------|------|
| **数据权限** | 控制可使用的 事件/事件属性、用户/用户属性 |
| **功能权限** | 控制可使用的分析模型、功能模块 |
| **脱敏设置** | 脱敏显示（中间打码）、禁止分组与筛选 |
| **成员管理** | 创建用户、管理角色、可分配角色 |

- 一个成员可被授予**多个角色**，最终权限取多角色**并集**
- 授权对象支持按 **账号** 或按 **账号属性（职务）** 授权
