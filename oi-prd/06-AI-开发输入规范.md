# AI 开发输入规范

## 1. 目的

本文件定义如何把本目录 PRD 直接交给 AI 开发工具使用，以减少歧义、避免过度发挥，并提高需求到实现的一致性。

## 2. AI 开发顺序

1. 先读 `00-产品愿景与边界.md`
2. 再读 `01-版本路线图.md`
3. 明确当前只开发哪个版本
4. 只实现当前版本 `In Scope` 内容
5. 对 `Out of Scope` 内容只预留扩展点，不提前实现

## 3. AI 必须遵守的边界

1. 不能跨版本偷做功能
2. 不能因为技术方便而弱化租户隔离和项目隔离
3. 不能把字段校验、权限校验、省略为“后续补充”
4. 不能将“页面可展示”误认为“能力可交付”
5. 不能省略审计、错误处理和异常状态

## 4. 开发拆解方式

每个版本都必须至少拆成以下交付物：

1. `domain model` 数据对象定义
2. `api spec` 接口定义
3. `page spec` 页面与交互定义
4. `permission matrix` 权限矩阵
5. `acceptance test` 验收测试
6. `seed data / demo data` 示例数据

## 5. AI 输出格式要求

当 AI 接到其中任一版本开发任务时，应输出以下内容：

1. 功能拆分清单
2. 数据表 / 存储结构设计
3. API 路由列表
4. 页面清单与状态流转
5. 测试用例清单
6. 风险点与待确认项

## 6. 推荐开发顺序

### V0.1

1. 认证与租户模型
2. 项目模型
3. 数据接收 API
4. 元数据入库与管理
5. 实时调试页
6. 权限与审计

### V0.2

1. 查询 DSL 与查询执行器
2. 事件分析
3. 属性分析
4. 漏斗分析
5. 留存分析
6. 概览系统

### V1.0

1. 用户细查与用户详情
2. 分群引擎
3. 路径 / 分布 / 间隔分析
4. 虚拟事件
5. 渠道链接
6. 预警中心

### V2.0

1. 标签引擎
2. 画像视图
3. 归因引擎
4. 指标平台
5. 智能诊断
6. 开放平台

## 7. 默认技术实现约束

如果没有额外说明，AI 开发时默认采用以下原则：

1. API 优先于前端实现
2. 所有查询接口支持分页、排序、筛选
3. 长耗时分析必须异步化
4. 前端必须处理空态、错误态、加载态
5. 资源删除默认软删除
6. 所有核心资源支持备注与创建人信息

## 8. 测试要求

### 必测类型

1. 单元测试
2. API 集成测试
3. 权限测试
4. 多租户隔离测试
5. 异常输入测试
6. 大分页和高并发基本压测

### 每个版本最少验收要求

| 版本 | 最少验收内容 |
| --- | --- |
| V0.1 | 接入链路 + 元数据治理 + 权限 |
| V0.2 | 四大分析模型 + 概览 |
| V1.0 | 分群闭环 + 用户细查 + 预警送达 |
| V2.0 | 标签复用 + 归因 + 指标统一口径 |

## 9. AI 不应擅自做的产品决策

1. 改变版本顺序
2. 删除权限模型
3. 合并租户与项目概念
4. 删除测试项目 / 正式项目概念
5. 把高级能力提前塞进低版本，造成 MVP 复杂度失控

## 10. 推荐给 AI 的任务模板

可直接将下面这段任务模板复制给 AI 开发工具：

```text
你现在要实现 OI Analytics Cloud 的指定版本功能。
请先阅读：
1. oi-prd/00-产品愿景与边界.md
2. oi-prd/01-版本路线图.md
3. oi-prd/对应版本 PRD
4. oi-prd/06-AI-开发输入规范.md

要求：
1. 仅实现当前版本 In Scope 内容
2. 先输出技术拆解，再实施开发
3. 输出数据模型、接口清单、页面清单、测试用例
4. 明确标出风险点和需确认项
5. 所有设计必须保留 SaaS 多租户、项目隔离、权限控制、审计日志
```
