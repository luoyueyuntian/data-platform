# 版本内迭代拆解

## 1. 说明

本文件用于把大版本继续拆成可直接开发的迭代批次。每个迭代都要求具备明确目标、开发范围和退出条件，避免 AI 或研发团队一次性吞下整版需求，导致实现失焦。

## 2. V0.1 迭代拆解

### Iteration V0.1-A

| 项目 | 内容 |
| --- | --- |
| 目标 | 打通基础 SaaS 组织、项目与数据接收链路 |
| 范围 | 登录、租户、项目、接收地址、token、HTTP 数据接收 API |
| 核心产出 | 可以创建项目并接收标准事件 |
| 退出条件 | 正确上报后可在库内查询到事件；项目隔离可验证 |

### Iteration V0.1-B

| 项目 | 内容 |
| --- | --- |
| 目标 | 让研发可自助验证接入结果 |
| 范围 | 数据接入引导、实时导入查询、错误样例、Debug 查询 |
| 核心产出 | 能快速定位“为什么没收到 / 为什么校验失败” |
| 退出条件 | 可筛选实时数据、查看原始 JSON、看到错误原因 |

### Iteration V0.1-C

| 项目 | 内容 |
| --- | --- |
| 目标 | 建立元数据治理与基础权限 |
| 范围 | 元事件、事件属性、用户属性管理；自由上报 / 强校验；成员角色 |
| 核心产出 | 数据可管、字段可控、权限可收口 |
| 退出条件 | 元数据可编辑；强校验能拦截错误数据；无权限用户不可访问项目数据 |

## 6. 每个迭代统一退出标准

1. 功能范围已完成，不留“界面先行、能力未实现”的假交付
2. 核心接口具备鉴权、参数校验、错误码和审计
3. 前端具备空态、错误态、加载态
4. 关键链路具备测试覆盖
5. 多租户与项目隔离通过验证

## 7. 推荐执行策略

1. 若团队资源有限，建议一次只开发一个迭代
2. 若使用多个 AI Agent 并行开发，按“页面 / 服务 / 数据模型 / 测试”分工，而不是按文件随意切分
3. 每个迭代结束后先做验收，再开始下一迭代，避免跨版本返工
